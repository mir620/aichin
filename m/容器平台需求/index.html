Rancher reqire

数据库密码有些是指定的，有些不是，不明白原因

镜像仓库间互转

  • 镜像仓库同步时，直接将头换

增加一个调试功能，临时保证容器可以运行起来

在服务下拉菜单中，增加一个“调试”，使用 启动一次 和 bash 属性

界面太乱，所有栈都在显示一个平面上，太乱

  • 显示的时候，可以将一些栈归为一个逻辑组，主页显示的时候，先显示”逻辑组“，打开才显示“逻辑组”里面的栈。

可以自定软件栈的权限，只有指定权限的用户才可以看

  • 在 tag 那里，设置权限，然后，只有这个用户可以看到

使用已有的栈布局直接生成 docker-compose 及 rancher-compose

  • 通过自动化实现直接提交 Git

如果已有 某个栈，再次部署时，不仅有部署，而且有更新的选项

  • upgrade service

通过配置文件 /conf 中需要替换的变量，来自动生成 docker-compose 中的 environment 里的变量

有时通过 racher-compose 自动化删除了栈后，容器还在，导致端口占用，只能手工删除容器

rancher-compose 有时无法删除栈，重试二次依然无效

业务不正常告警

业务运行所有的日志（包括编译日志）可以通过系统查看

  • 重复日志显示总条数

如果服务只创建不启动，升级时会报错

服务已经成功，但软件栈还是显示红色

standalone 容器会自动启动，从而占用端口，导致部署失败

  • 定期清理 standalone 的容器

1、应用商店服务部署启动时，如果有多个分类，需要有明确的分隔，比如数据库1，数据库2.
2、云平台需要支持镜像版本的导出和导入（一键部署指定版本）。
3、镜像仓库之间的数据可以直接自动转换。
4、镜像版本的管理与配套关系的系统。
